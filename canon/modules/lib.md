# Module: src/lib (shared runtime)

> Generated by Cicadas at arcade-buildout initiative close.

## Purpose

The frozen foundation every game builds on: wake/sleep/fullscreen arbitration and the sole owner of document-level input (`hub.ts`), the per-game contract shape (`cartridge.ts`), tolerant persistence (`storage.ts`), audio (`audio.ts`), pixelated canvas setup (`screen.ts`), a fixed-timestep game loop (`loop.ts`), and touch gesture recognition (`input.ts`). Frozen after `feat/foundation` â€” any change here must be broadcast as a Cicadas Signal so parallel game work isn't silently invalidated.

- **Dependencies**: none outside the module â€” this is the leaf layer everything else imports from. No runtime npm packages.

## Key Decisions

- **`hub.ts` is the only code allowed to call `document.addEventListener` for keyboard/mouse, and the only caller of `preventDefault` on nav keys.** This is the PRD's #1 flagged bug class (input leaking across games / stealing scroll) and is enforced by convention + code review, not by a lint rule.
- **Document listeners attach lazily** on first `Hub.register()` call â€” safe to import `hub.ts` under Vitest/node where `document` may not exist yet.
- **`loop.ts` decouples update from render**: fixed-step `update(dt)` (default 60 Hz) via an accumulator, `render(alpha)` once per animation frame, `maxDeltaMs` (default 250) clamps a background-tab return from unleashing thousands of catch-up updates.
- **`screen.ts` scales the backing store by an integer DPR** (capped, default 2; pass `dprCap: 1` to reproduce the original mockup's exact pixel grid) so low-res pixel art stays crisp on hidpi without going soft.
- **`input.ts` never attaches to `document`** â€” element-scoped only. Scroll-vs-swipe ownership is decided by CSS `touch-action` (`.gcard.active .screen { touch-action: none }`), not by fighting over `preventDefault`.
- **`storage.ts` is tolerant by design**: `get()` takes an optional type-guard validator and falls back + `console.warn`s on corrupt or unexpected-shape data rather than throwing â€” there is no server to fall back to, so a corrupt localStorage value must never crash the page.
